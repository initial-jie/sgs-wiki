# SGS-Wiki 线下房间协议 v0.1

> 状态:草案已拍板核心决策,进入 node 模拟阶段。
> 目标:把 12 个单机线下工具接入一个"多人房间",零后端服务器成本、完全离线可用(除房间信令),保密模型不弱于现状。

## 0. 术语

- **DO** = Cloudflare **Durable Object**。一个房间 = 一个 DO 实例:单点权威、内存态、持有所有 WebSocket 连接并广播。**不是** Data Object。
- **座位(Seat)** = 房间里的一个位置,是身份的唯一载体。
- **设备(Device)** = 一台手机上的一个浏览器连接,是可替换的外壳。
- **工具(Tool)** = 现有 `tools/*.html` 里的某个武将线下工具。

## 1. 六条地基原则

1. **DO 是唯一权威**(single source of truth),所有手机都是客户端;开房者不特殊,只是第一个连接的人。
2. **房间座位是唯一真相**,工具内自带的花名册(貂蝉/董昭/孙权/吕布)一律绑定到房间座位号,不再各管各的。
3. **保密在 DO 端按请求者身份过滤后才下发**,绝不"发完整状态到前端再隐藏"(否则抓包即作弊)。
4. **座位 ≠ 设备**(解耦):一个座位可被任意设备认领;一个设备可认领多个座位。这是 fallback 的地基。
5. **工具业务逻辑不动**,在外面套"连房间壳"。
6. **DO 只管数据 + 可见性过滤 + 广播,不搬业务 reducer**(见决策①=方案 a)。

## 2. 核心模型

```
Room  (= 一个 DO 实例, 4 位短码 roomCode)
 └─ Seat[1..N]  { seatNo, general, holderDevices[], toolState }
Device (deviceId 存 localStorage, 可断线重连)  认领 0..N 个座位
```

"玩家"约等于"座位":身份挂在座位上,设备换了/被代持了,座位身份不变。

## 3. 座位-设备解耦与 fallback(传手机的自然退化)

fallback 不是额外功能,而是"一个设备可认领多个座位"的自然涌现:

- **单座位代持**:某设备 `claimSeat` 一个没电玩家的座位,替他操作。因为它认领了该座位,DO 就把该座位完整状态发给它 —— 等价于"帮人传手机记牌的人本来就看得到那份牌",保密等同现状。
- **多座位集中**:一台设备认领多个座位,屏幕上切换"当前操作座位" = 局部传手机。
- **全单机**:开房设备认领全部座位 = 完全等于现在的单机工具,只是状态存在 DO,别人有电了可随时 `claimSeat` 接管。

代码上不保留第二套"传手机工具";UI 上显式提供「代持座位 / 切换当前操作座位」入口即可。

## 4. 可见性原语(每个工具声明 spec,DO 通用过滤)

| 原语 | 含义 | 使用者 |
|---|---|---|
| `public` | 对所有座位可见 | 9 个公开台账工具 |
| `ownerOnly(owners, unlockWhen)` | 仅 owners 座位可见,直到 `unlockWhen` 成立后转全公开 | 南华(owners=南华[+受术座位], unlock=该天书已发动)、孙权(unlock=已揭示) |
| `secretHolding` | 多座位各写各的,**明细仅登记者本人(及代持设备)可见**;聚合明细**仅 DO 系统内部可读用于结算**,任何座位(含吕布本人)都看不到他人明细;**数量对全场公开** | 吕布"登记初始牌为炁" |
| `ownerSeatOnly(seat)` | 仅该座位本人可见,他人只见数量 | 吕布"夺获的炁"、袁姬"镜花/水月牌名"、董昭"先略牌名" |
| `secretPick` | 键值对象 `{[座位]:{holder,effect,revealed}}`,每份自带 `revealed`;翻开前**仅本人(或代持)可见内容**,他人只见 `{holder,hidden}` 占位(能数出进度、不知内容,连孙权也偷看不到);`reveal` 时 DO 原子翻开+结算 | 魔孙权"权御暗选"(密封同时揭示) |

DO 用 `工具数据状态 + spec + 请求设备认领的座位集合` 过滤出**每台设备各自的视图**。新工具接入只声明 spec,DO 无需改动。

> **实现现状**:上述原语均已在 `prototype/shared/room-logic.mjs` 的 `filterState` 实现并被 12 工具使用。`ownerOnly` 为南华用的数组形态(每册自带 owners+revealed);魔孙权暗选另用键值形态 `secretPick`。

## 5. 消息协议(WebSocket)

上行(客户端 → DO):
```
createRoom{}                              → {roomCode}
joinRoom{roomCode, deviceId}
claimSeat{seatNo} / releaseSeat{seatNo}
setGeneral{seatNo, generalId}
action{targetSeat, bySeat, toolAction}    ← 对某座位工具执行一个操作
```

`action` 的 DO 校验规则:
- 普通写(操作自己/代持的座位):`bySeat` 必须 ∈ 该设备认领的座位集合。
- **跨座位私密写**(如吕布"别人登记初始牌"):允许任意座位往 `targetSeat` 写属于 `bySeat` 自己的那份;读取时受该字段的 `privateAggregate` 保护。

下行(DO → 客户端):
```
roomState{ seats:[... 按本设备身份过滤后 ...], youHold:[认领的座位号] }   ← 每台设备内容不同
error{code, msg}
```

## 6. 状态权威 = 方案 a(已拍板)

- DO 存**纯数据状态**(JSON) + 做**通用 JSON 应用 + 可见性过滤 + 广播**。
- 业务 reducer 仍在前端(复用现有工具全部逻辑)。认领某座位的设备是该座位状态的唯一写入者,故无并发争抢。
- 好处:复用现有 12 个工具全部前端逻辑,DO 保持通用,契合"工具逻辑不动"。
- **针对性例外(保密结算下沉 DO)**:当某操作需要读取"对操作者本人保密"的数据时,其结算逻辑**必须在 DO 端执行**,前端只收结果。典型:吕布夺炁——各玩家暗置的炁明细吕布不可见,"抽哪张"由 DO 按规则决定,吕布前端仅收到夺获结果;若在前端算,吕布就会收到全部暗牌 = 抓包作弊。这类"保密结算"是方案 a 的少数例外,不影响其余公开台账工具的 reducer 留在前端。

## 7. 决策记录

- ① 状态权威:**方案 a**(DO 存数据+过滤,reducer 在前端)。
- ② roomCode:**4 位数字**(线下口报方便;scale 小,日后好改)。
- ③ 实施顺序:**吕布 → 南华 → 其余(除魔孙权)→ 魔孙权**。

## 8. 环境约束

- 无外网;真实 WebSocket 联通须本地/部署后验证。
- 协议逻辑用 node 模拟双端(多设备)跑测试(见 `prototype/` 下模拟脚本)。
- GitHub Pages 的 HTTPS 满足摄像头/PWA;房间信令走 Cloudflare Workers + DO,零常驻后端成本。
